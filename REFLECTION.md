# REFLECTION.md

## 1. Почему эта задача показалась мне интересной (личная мотивация)

Задача привлекла меня тем, что находится на стыке трёх вещей: автоматизации, работы с реальными API и систем принятия решений. Вместо обычного скрипта, выполняющего фиксированную операцию, я создавал агента, способного читать репозиторий, понимать запрос пользователя и решать, какие изменения в коде нужно внести. Это гораздо ближе к тому, как работают настоящие инструменты разработчика (например, GitHub Copilot или внутренние дев-боты), которые сейчас активно разрабатываются и вдохновляют многих людей.

Особенно захватывающим было то, что система недетерминирована. Один и тот же ввод может порождать разные планы в зависимости от ответа модели. Это привносит неопределённость, которая заставляет думать о надёжности, обработке ошибок и структуре кода гораздо глубже, чем в типичных проектах.

Инструмент такого рода полезен разработчикам, особенно новичкам или командам, которые хотят автоматизировать повторяющиеся изменения кода (генерация шаблонного кода, исправление простых багов, расширение небольших проектов). Это также отличный учебный проект, потому что он вынуждает мыслить конвейерно: ввод → анализ → планирование → выполнение → проверка.

## 2. Где агент давал сбои (точки отказа)

В процессе разработки обнаружилось несколько проблем, которые связаны скорее с требовниями к разработке(отсутствие токенов, малое время на разработку и отлаживание кода):

**1. LLM возвращает невалидный или неструктурированный вывод**  
Поначалу модель часто возвращала текст на естественном языке вместо строгого JSON. Из-за этого json.loads() может падать с ошибкой. Даже при жёстких промптах модели иногда игнорируют правила форматирования.

**2. Нестабильность API и ограничения частоты запросов**  
Использование бесплатных тарифов моделей приводило к частым ошибкам доступа, превышения числа обращений к серверу и так далее. Даже логика повторных попыток не помогала, потому что проблема была не во временном сетевом сбое, а в исчерпании лимитов бесплатных моделей.

**3. Разное поведение моделей**  
Разные модели вели себя по-разному: одни хорошо следовали правилам формирования JSON, другие выдавали ответы в разговорном стиле. Это делало конвейер нестабильным и усложняло отладку.

Эти сбои в целом очевидные и показывают, что системы на основе LLM требуют защитного проектирования с множественными проверками: нельзя исходить из предположения, что модель всегда будет возвращать что требуется.

## 3. Что бы я улучшил в будущем

Если бы у меня было больше времени, я бы сосредоточился на том, чтобы сделать систему более стабильной для промышленной эксплуатации:

- Добавить **аварийное переключение между моделями** (при отказе одного API переходить на другое)
- Реализовать **валидатор JSON + шаг автовосстановления** перед парсингом
- Кешировать промежуточные результаты (например, сводку по репозиторию) для снижения числа обращений к API
- Ввести **цикл повторного планирования** до тех пор, пока не будет получен валидный JSON
- Добавить лёгкий «симулятор выполнения» для предпросмотра изменений перед реальной записью на GitHub

```
           │
           ▼

│   Слой GitHub API     │
│ (список / чтение файлов) │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐

└──────────┬───────────┘
           │
           ▼

│      LLM: Сводка репозитория     │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│  LLM: Генератор плана (STRICT JSON) │
└──────────────┬──────────────────┘
               │
               ▼
     ┌─────────────────────┐
     │  Для каждого пункта   │
     │       плана:          │
     └─────────┬─────────────┘
               │
               ├──▶ Чтение файла (если изменение)
               │
               ├──▶ LLM: Генератор кода
               │
               ├──▶ Генератор диффа
               │
               ├──▶ Подтверждение пользователем
               │
               └──▶ GitHub API записи
                        │
                        ▼

│   Итоговый обновлённый репозиторий │
└─────────────────────────────────┘
```
## 4. Вывод
По своей сути агентная система представляет собой последовательный конвейер, где LLM выступает в роли планировщика и генератора кода, а Python отвечает за управление последовательстью шагов LLM, валидацию и взаимодействие с внешними системами (GitHub и API).
